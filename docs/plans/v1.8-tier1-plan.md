## Task A: Budget Detail View - Scope to Selected Budget Only (#20)

### Problem

When clicking a specific budget (e.g., "Groceries - Feb 2026") to view its ledger, the service fetches expenses by **category + month** rather than by **budgetId**. This means if a user has two budgets under the same category (e.g., "Car Insurance" and "Life Insurance" both under "Insurance"), clicking either one shows **all** expenses for that category - not just the ones linked to that specific budget.

The status cards (Budget Limit, Total Spent, Remaining, Daily Pace) also reflect the combined category total instead of the individual budget envelope.

### Accountant's Perspective

Envelope budgeting requires strict isolation. Each envelope (budget) tracks its own spending. Merging transactions across envelopes of the same category defeats the purpose - you can't tell if "Car Insurance" is on track if it's mixed with "Life Insurance" expenses.

### Current Behavior

**File:** `server/modules/budget/budget.service.ts` → `getBudgetWithExpenses()`

```typescript
// Current: Fetches ALL expenses for the budget's category in that month
const expenses = await prisma.expense.findMany({
  where: {
    userId,
    categoryId: budget.categoryId,  // Too broad
    date: { gte: startOfMonth, lte: endOfMonth },
  },
});
```

### Required Changes

**File:** `server/modules/budget/budget.service.ts`

Change the expense query in `getBudgetWithExpenses()` to filter by `budgetId` instead of `categoryId`:

```typescript
// Fix: Fetch only expenses linked to THIS specific budget
const expenses = await prisma.expense.findMany({
  where: {
    userId,
    budgetId: budget.id,  // Scoped to this envelope
    date: { gte: startOfMonth, lte: endOfMonth },
  },
  orderBy: { date: 'asc' },
  include: { account: true },
});
```

**Fallback consideration:** Some expenses may have been created without a `budgetId` (before the budget-linking feature existed). Add a secondary section or note in the UI showing "Uncategorized expenses in this category" for transparency:

```typescript
// Optional: Also fetch unlinked expenses in same category for reference
const unlinkedExpenses = await prisma.expense.findMany({
  where: {
    userId,
    categoryId: budget.categoryId,
    budgetId: null,  // Not linked to any budget
    date: { gte: startOfMonth, lte: endOfMonth },
  },
  orderBy: { date: 'asc' },
  include: { account: true },
});
```

### Files to Modify

## Task B: Budget Dropdown - Smart Grouped Selection (#21)

### Problem

The expense form's budget dropdown shows **all budgets across all months** in a flat list. When creating a February expense, the user sees January, March, and other months' budgets mixed together. This can lead to accidentally linking an expense to the wrong month's budget, corrupting budget tracking.

### Accountant's Perspective

Budget periods are sacred. A February grocery expense should default to the February grocery budget. However, there are legitimate cases for cross-month linking: late categorization of expenses, corrections, or expenses that straddle month boundaries. The system should **guide** toward the correct month while **allowing** flexibility.

### UX Design: Sectioned Dropdown with Smart Defaults

The dropdown uses grouped sections with clear visual hierarchy. The expense date's month is the **primary section** (prominent, no month labels needed). Previous months are a **secondary section** (de-emphasized, includes month labels for disambiguation). This handles 95% of cases immediately while keeping 5% edge cases accessible.

#### Dropdown Layout

```text
┌─────────────────────────────────────────────┐
│  Link to a budget (optional)            v   │
├─────────────────────────────────────────────┤
│  o  No budget                               │
│                                             │
│  -- February 2026 -----------------------   │  <- Matches expense date
│  o  Groceries              (P4,500 left)    │  <- No month label needed
│  o  Dining Out             (P1,200 left)    │
│  o  Transportation           (P800 left)    │
│  o  Utilities              (P2,000 left)    │
│                                             │
│  -- Previous Months ----------------------  │  <- Secondary section
│  o  Groceries - Jan 2026      (P0 left)    │  <- Includes month label
│  o  Dining Out - Jan 2026   (P300 left)    │
│  o  Transportation - Dec 2025 (P150 left)  │
└─────────────────────────────────────────────┘
```

#### Design Decisions

1. **Current month budgets omit the month label** - The section header already says "February 2026". Less noise, faster scanning.
2. **Previous month budgets include the month label** - Essential for disambiguation: "Groceries - Jan 2026" vs "Groceries - Dec 2025".
3. **Previous months styled with `text-muted-foreground`** - Visually secondary. User's eye goes to the primary section first.
4. **Previous months sorted newest-first** - Most likely need is last month, not 6 months ago.
5. **Limited to 3 months back** - Anything older is almost certainly a mistake. Keeps dropdown manageable.
6. **Date change updates grouping dynamically** - If user picks Jan 15, January budgets become the primary section and February moves to "Other Months".

#### Interaction Flow

```text
User opens expense form
  -> Date defaults to today (Feb 6, 2026)
  -> Budget dropdown: Feb 2026 budgets on top, Jan/Dec/Nov below
  -> User picks "Groceries" (Feb 2026) -- most common path

User changes date to Jan 20, 2026
  -> Selected budget is cleared (month changed)
  -> Budget dropdown: Jan 2026 budgets now on top
  -> Feb 2026 budgets move to "Other Months"
  -> User picks "Groceries" (Jan 2026)
```

**File:** `app/(authenticated)/expense/page.tsx`


**File:** `components/modules/expense/ExpenseForm.tsx`

### Required Changes

#### Step 1: Split budgets into current month and previous months

**File:** `components/modules/expense/ExpenseForm.tsx`

```typescript
const selectedDate = form.watch('date');

const { currentMonthBudgets, previousMonthBudgets } = useMemo(() => {
  if (!selectedDate) return { currentMonthBudgets: budgets, previousMonthBudgets: [] };

  const expMonth = selectedDate.getMonth();
  const expYear = selectedDate.getFullYear();

  const current: BudgetWithCategory[] = [];
  const previous: BudgetWithCategory[] = [];

  // 3-month lookback limit
  const threeMonthsAgo = new Date(expYear, expMonth - 3, 1);

  for (const b of budgets) {
    const bDate = new Date(b.month);
    const bMonth = bDate.getMonth();
    const bYear = bDate.getFullYear();

    if (bMonth === expMonth && bYear === expYear) {
      current.push(b);
    } else if (bDate >= threeMonthsAgo && bDate < new Date(expYear, expMonth, 1)) {
      previous.push(b);
    }
    // Budgets older than 3 months or in the future are excluded
  }

  // Sort previous: newest month first, then by name within month
  previous.sort((a, b) => new Date(b.month).getTime() - new Date(a.month).getTime());

  return { currentMonthBudgets: current, previousMonthBudgets: previous };
}, [selectedDate, budgets]);
```

#### Step 2: Reset budget selection when date month changes

```typescript
const prevMonthRef = useRef<string | null>(null);

useEffect(() => {
  if (!selectedDate) return;
  const monthKey = `${selectedDate.getFullYear()}-${selectedDate.getMonth()}`;
  if (prevMonthRef.current && prevMonthRef.current !== monthKey) {
    form.setValue('budgetId', undefined);
    // Category lock is released since budget was cleared
  }
  prevMonthRef.current = monthKey;
}, [selectedDate, form]);
```

#### Step 3: Render sectioned dropdown

```tsx
<SelectContent>

  {/* Primary: Current month budgets */}

        {format(selectedDate, 'MMMM yyyy')}
      </SelectLabel>
      {currentMonthBudgets.map((budget) => (
        <SelectItem key={budget.id} value={budget.id}>
          <span className='flex items-center gap-2'>
            <span>{budget.name}</span>
            <span className={cn('text-xs font-medium', colorByRemaining(budget))}>
              ({formatCurrency(budget.remaining, { decimals: 0 })} left)
            </span>
          </span>
        </SelectItem>
      ))}
    </SelectGroup>
  )}

  {/* Empty state: no budgets for current month */}
  {currentMonthBudgets.length === 0 && (
    <SelectGroup>
      <SelectLabel className='text-xs text-muted-foreground italic'>
        No budgets for {format(selectedDate, 'MMMM yyyy')}
      </SelectLabel>
    </SelectGroup>
  )}

  {/* Secondary: Previous months (de-emphasized) */}
  {previousMonthBudgets.length > 0 && (
    <SelectGroup>
      <SelectLabel className='text-xs font-semibold uppercase tracking-wider text-muted-foreground'>
        Previous Months
      </SelectLabel>
      {previousMonthBudgets.map((budget) => (
        <SelectItem key={budget.id} value={budget.id} className='text-muted-foreground'>
          <span className='flex items-center gap-2'>
            <span>{budget.name} - {format(new Date(budget.month), 'MMM yyyy')}</span>
            <span className={cn('text-xs font-medium', colorByRemaining(budget))}>
              ({formatCurrency(budget.remaining, { decimals: 0 })} left)
            </span>
          </span>
        </SelectItem>
      ))}
    </SelectGroup>
  )}
</SelectContent>
```

## Task C: Account Type Separation - UI Grouping (#12)

### Problem

The accounts page (`/accounts`) shows all accounts in a **flat list** with no grouping. Users with many accounts (bank, cash, credit cards, loans, emergency fund, savings goals) see them all mixed together. The dashboard groups them into "Liquid Assets" and "Liabilities" but this is inconsistent with how the account form already categorizes them into 3 groups (Asset, Liability, Fund).

### Accountant's Perspective

Financial statements always separate: Current Assets, Fixed Assets, Current Liabilities, Long-term Liabilities, and Equity/Reserves. For a personal finance app, the minimum separation should be:
- **Liquid Assets** (BANK, CASH) - immediately available money
- **Savings & Investments** (SAVINGS, INVESTMENT) - available but not for daily spending
- **Liabilities** (CREDIT, LOAN) - money owed
- **Funds & Goals** (EMERGENCY_FUND, FUND) - reserved/goal-based money

This gives users a clear mental model of their financial position at a glance.

### Current Behavior

**Accounts page:** Flat DataTable, no grouping
**Dashboard:** Binary split (`isLiability` true/false) + separate Funds section
**Account Form:** Already groups into Asset/Liability/Fund in the type selector

### Required Changes

#### Step 1: Define account classification utility

**New file:** `lib/account-utils.ts`

```typescript
export type AccountClass = 'liquid' | 'savings' | 'liability' | 'fund';

export const ACCOUNT_CLASS_MAP: Record<AccountType, AccountClass> = {
  BANK: 'liquid',
  CASH: 'liquid',
  SAVINGS: 'savings',
  INVESTMENT: 'savings',
  CREDIT: 'liability',
  LOAN: 'liability',
  EMERGENCY_FUND: 'fund',
  FUND: 'fund',
  TITHE: 'fund',
};

export const ACCOUNT_CLASS_META: Record<AccountClass, { label: string; icon: string; color: string }> = {
  liquid: { label: 'Liquid Assets', icon: 'Wallet', color: 'emerald' },
  savings: { label: 'Savings & Investments', icon: 'PiggyBank', color: 'blue' },
  liability: { label: 'Liabilities', icon: 'CreditCard', color: 'red' },
  fund: { label: 'Funds & Goals', icon: 'Target', color: 'violet' },
};

export function groupAccountsByClass(accounts: Account[]) {
  return {
    liquid: accounts.filter(a => ACCOUNT_CLASS_MAP[a.type] === 'liquid'),
    savings: accounts.filter(a => ACCOUNT_CLASS_MAP[a.type] === 'savings'),
    liability: accounts.filter(a => ACCOUNT_CLASS_MAP[a.type] === 'liability'),
    fund: accounts.filter(a => ACCOUNT_CLASS_MAP[a.type] === 'fund'),
  };
}
```
