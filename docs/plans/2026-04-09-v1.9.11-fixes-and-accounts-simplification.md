## Task 4: Opening balance — synthetic ledger row via dedicated column

**Context:** `server/modules/account/account.service.ts:21-74`. The current `createAccount` creates an INCOME ledger row under the `💰 Opening Balance` category for asset accounts, and nothing at all for liabilities. Two problems:

1. **Report/KPI pollution.** `💰 Opening Balance` Income rows are not excluded from any aggregate in `server/modules/report/report.service.ts`, `server/modules/transaction/transaction.service.ts`, or `server/modules/dashboard/dashboard.service.ts`. They inflate monthly reports, category breakdowns, savings rate, and health score for the onboarding month. Extending this to liabilities would make it worse because expense-side pollution distorts budget utilization.

2. **Ledger gap for liabilities.** Liability accounts have no opening ledger row at all, so their running balance starts from an invisible baseline (this is the mysterious ₱300 the user saw).

**Fix — the synthetic row approach.** Add an `openingBalance Decimal?` column to the `Account` model. `createAccount` writes the starting balance to both `balance` and `openingBalance`. The existing Income/Expense row creation for openings is removed entirely. `getAccountWithTransactions` prepends a synthetic `{ type: 'OPENING', amount: openingBalance }` row at the bottom of the ledger so the reverse-walk running balance lands on the opening balance as the first entry. `AccountLedger.tsx` renders OPENING rows with muted/italic styling and a distinct badge.

**Why this is structurally better:**
- No Income/Expense row exists → report/KPI pollution becomes impossible forever.
- Also fixes the pre-existing asset-side pollution for new accounts.
- Eliminates the "don't use `ExpenseService.createExpense` or you'll double-decrement" footgun.
- The new column is isolated to one service + one display layer.

**Scope of migration:** The schema migration only adds the column (nullable, default null). We do NOT backfill historical `💰 Opening Balance` Income rows on existing accounts — user is restarting their data, and other users' historical rows remain unchanged until they recreate accounts. This is documented as a known limitation, not a regression.

- Modify: `prisma/schema.prisma` — add `openingBalance` column to `Account`

**Step 4: Prepend synthetic OPENING row in `getAccountWithTransactions`**

Read `account.service.ts:190-259` to understand the current shape of the function: it fetches the account, aggregates INCOME + EXPENSE + TRANSFER + PAYMENT rows into a `transactions` array, sorts by date descending, then reverse-walks to compute `runningBalance` for each.

At the end — AFTER the reverse-walk is complete and each transaction has `runningBalance` populated — append a synthetic OPENING row to the `transactions` array IF `account.openingBalance !== null`:

```ts
if (account.openingBalance !== null) {
    transactions.push({
        id: `opening-${account.id}`,
        type: 'OPENING' as const,
        amount: account.openingBalance,
        description: 'Opening Balance',
        date: account.createdAt,
        categoryName: null,
        relatedAccountName: null,
        runningBalance: account.openingBalance,
    });
}
```

Notes:
- Push, don't unshift: transactions are sorted newest-first, so the oldest row is at the end.
- The `id` prefix `opening-` is synthetic and distinct from any real transaction UUID.
- `runningBalance` for the OPENING row equals `openingBalance` itself — the balance state immediately after account creation and before any real transaction.
- Do NOT include the OPENING row in the reverse-walk loop. The reverse walk starts from `account.balance` and derives the running balance after each real transaction; by the time it reaches the earliest real transaction, the running balance before it should equal `openingBalance`. That's your reconciliation check (see Step 7 manual test).

                const subtotal = groupAccounts.reduce((sum, a) => sum + Number(a.balance), 0);
                const isLiability = cls === 'liability';

                return (

                                    meta.color === 'blue' && 'text-blue-600 dark:text-blue-400',
                                    meta.color === 'red' && 'text-red-600 dark:text-red-400'
                                )}
                            >
                                <Icon className='h-4 w-4' />
                                {meta.label}
                                <span className='opacity-70'>({groupAccounts.length})</span>
                            </h3>
                            <span
                                className={cn(
                                    'text-sm font-bold tabular-nums',
                                    isLiability ? 'text-red-600 dark:text-red-400' : 'text-foreground'
                                )}
                            >
                                {formatCurrency(subtotal)}
                            </span>
                        </div>

                        <div className='rounded-md border bg-card'>
                            <Table className='table-fixed'>
                                <TableHeader>
                                    <TableRow>
                                        <TableHead className='w-[40%]'>Name</TableHead>
                                        <TableHead className='hidden sm:table-cell w-[20%]'>Type</TableHead>
                                        <TableHead className='w-[25%] text-right'>Balance</TableHead>
                                        <TableHead className='w-[15%] text-right'>Actions</TableHead>
                                    </TableRow>
                                </TableHeader>
                                <TableBody>
                                    {groupAccounts.map((account) => (

                                    ))}
                                </TableBody>
                            </Table>
                        </div>
                    </section>
                );
            })}
        </div>
    );
}
